# US-001: Harden Auth and Configuration Defaults

**Status**: In progress — concrete hardening implemented, remaining risks documented below.

## Changes Implemented

| # | Change | Files | Impact |
|---|--------|-------|--------|
| 1 | **Production block on default `app_password`** — pydantic validator raises `ValueError` when `APP_ENV != "development"` and `APP_PASSWORD` is still `"changeme"`. App refuses to start. | `backend/app/config.py` | Prevents silent deployment with known password |
| 2 | **Startup warning for insecure defaults** — `_check_insecure_config()` in `main.py` logs prominent warnings at boot for `app_password` and `secret_key`. | `backend/app/main.py` | Operators see the risk in logs immediately |
| 3 | **Pydantic validator warns on default `secret_key`** — emits a warning at config-load time even in development. | `backend/app/config.py` | Early signal that `SECRET_KEY` needs rotation |
| 4 | **Frontend clears stored password on 401** — `apiFetch` now calls `clearPassword()` when the backend returns 401, preventing a wrong/stale password from persisting in `localStorage`. | `frontend/lib/api.ts` | Reduces user confusion + limits credential persistence |
| 5 | **Export `clearPassword`** — new public helper for explicit credential clearing (e.g. logout). | `frontend/lib/api.ts` | Enables future logout flow |
| 6 | **Added `APP_ENV` to `.env.example` and `docker-compose.yml`** | `.env.example`, `docker-compose.yml` | Operators know the knob exists |
| 7 | **Documented localStorage XSS risk in `api.ts` header comment** | `frontend/lib/api.ts` | Future developers see the trade-off |

## Remaining Risks (Severity-Ranked)

### CRITICAL

| ID | Finding | Impacted Files | Recommended Action |
|----|---------|---------------|-------------------|
| C-1 | **Password stored in `localStorage`** — accessible to any JS in the page origin. If XSS is introduced (via a dependency, user input rendering, or the coaching chat), the credential is exfiltrated in one line. | `frontend/lib/api.ts:38`, `frontend/components/settings/SettingsView.tsx:150` | Migrate to an `HttpOnly` cookie set by the backend after a login endpoint, or use a short-lived session token with refresh rotation. This is the single most impactful security improvement available. |
| C-2 | **`secret_key` unused but defaults to known value** — currently not used for signing or JWTs, but if any future feature (sessions, signed cookies, API tokens) consumes it, the default is catastrophically weak. | `backend/app/config.py:37` | Enforce minimum length/randomness at the validator level in addition to the warning. Consider generating a random key on first run if none is provided. |

### HIGH

| ID | Finding | Impacted Files | Recommended Action |
|----|---------|---------------|-------------------|
| H-1 | **Proxy route forwards all headers blindly** — the Next.js proxy (`[...path]/route.ts`) forwards the `Authorization` header from the client to the backend without inspection or rate limiting. A compromised frontend could send arbitrary requests. | `frontend/app/api/proxy/[...path]/route.ts` | Add rate limiting at the proxy level. Consider validating that the proxied request path is on an allowlist. |
| H-2 | **No brute-force protection on `/health` is exposed (unauthenticated)** — the health endpoint is intentionally open, but combined with H-1 the backend has no request-level rate limiting, making online password guessing possible. | `backend/app/main.py:57-59` | Add a simple in-memory rate limiter (e.g. `slowapi`) to `verify_password` or at the middleware level. |
| H-3 | **Database credentials hardcoded in defaults** — `database_url` default is `paceforge:paceforge@localhost:5432/paceforge`. If the container is network-accessible and the `.env` is missing, Postgres accepts these defaults. | `backend/app/config.py:16`, `docker-compose.yml:8-9` | Remove the hardcode; require `DATABASE_URL` to be explicitly set. The docker-compose defaults for local dev are fine, but the Python code should not silently fall back. |

### MEDIUM

| ID | Finding | Impacted Files | Recommended Action |
|----|---------|---------------|-------------------|
| M-1 | **CORS allows `localhost:3001` by default** — not harmful in production (origins would be overridden), but in dev the second origin expands the attack surface slightly. | `backend/app/config.py:66` | Low priority; acceptable for self-hosted. |
| M-2 | **Proxy strips `WWW-Authenticate` but doesn't add security headers** — no `X-Content-Type-Options`, `X-Frame-Options`, or `Content-Security-Policy` are set on proxied responses. | `frontend/app/api/proxy/[...path]/route.ts` | Add common security headers to the proxy response. |
| M-3 | **No HTTPS enforcement** — the app works over HTTP in local dev and has no HSTS or redirect-to-HTTPS behavior. For a self-hosted app behind a reverse proxy this is usually fine, but worth noting. | N/A (infra) | Document that a reverse proxy (Caddy, nginx) should terminate TLS. |

### LOW

| ID | Finding | Impacted Files | Recommended Action |
|----|---------|---------------|-------------------|
| L-1 | **`verify_password` doesn't log failed attempts** — no audit trail for brute-force attempts. | `backend/app/main.py:32-43` | Add a warning-level log on auth failure (without logging the attempted password). |
| L-2 | **Username in Basic auth is ignored** — `verify_password` returns `credentials.username` but it's never checked. Any username is accepted. | `backend/app/main.py:43` | For single-user app this is fine. If multi-user is ever added, this becomes a vulnerability. |
