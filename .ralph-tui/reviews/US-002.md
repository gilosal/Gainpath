# US-002: Stabilize Session Completion Side Effects

## Review Date: 2026-04-18

## Summary
End-to-end review of the session completion pipeline: `PATCH /sessions/{id}` → `_on_session_completed` → PR detection → streak update → XP grant → achievement check → post-workout coaching.

---

## Findings

### H-1 — Background task uses request-scoped DB session (FIXED)
**Severity:** High  
**Files:** `backend/app/routers/sessions.py:133`  
**Impact:** The `_on_session_completed` background task received the `db` dependency-injection session from the request. FastAPI's dependency lifecycle closes this session after the response is sent, but `BackgroundTasks` execute after the response. This creates a race condition: the DB session is likely closed by the time the task runs, causing `DetachedInstanceError` or `SessionClosed` exceptions that silently kill the entire completion pipeline — no PRs, no streak, no XP, no coaching.

**Fix:** Changed `_on_session_completed` to create its own `SessionLocal()` session with a `try/finally: db.close()` wrapper, matching the pattern already used in the scheduler jobs in `main.py`.

### H-2 — No error isolation between completion steps (FIXED)
**Severity:** High  
**Files:** `backend/app/routers/sessions.py:22-51`  
**Impact:** A crash in any step (e.g., PR detection) would short-circuit all remaining steps (streak, XP, achievements, coaching). Common failure modes: AI timeout during coaching, DB constraint error in PR detection, or any unhandled exception.

**Fix:** Wrapped each step in its own `try/except` with `log.exception(...)`. Steps that fail are logged and skipped; remaining steps proceed normally.

### M-1 — `completed_at` not auto-set on completion (FIXED)
**Severity:** Medium  
**Files:** `backend/app/routers/sessions.py:153-155`  
**Impact:** The `early_bird` achievement and coaching prompt both reference `session.completed_at`. If the client doesn't explicitly send `completed_at` in the PATCH payload, it remains `null`. This means the early-bird achievement can never fire, and coaching feedback lacks a completion timestamp.

**Fix:** Added logic in `update_session` to auto-set `completed_at = datetime.utcnow()` when transitioning to `"completed"` status, if not already set.

### M-2 — No idempotency guard against double-completion
**Severity:** Medium  
**Files:** `backend/app/routers/sessions.py:149-161`  
**Impact:** Two concurrent PATCH requests could both see `was_completed = False` and fire the completion chain twice, resulting in duplicate XP grants and double streak/achievement processing. The `was_completed` check is evaluated before the commit, so it's not fully race-proof under concurrent load.

**Status:** Not fully fixed — the `was_completed` guard prevents the common case of a single client re-sending. A full fix would require a database-level status constraint or a deduplication mechanism (e.g., idempotency key or `SELECT ... FOR UPDATE`). This is a lower priority since the app is currently single-user.

### L-1 — `xp` defaults to 0 if `xp_for_session`/`grant_xp` fails
**Severity:** Low  
**Files:** `backend/app/routers/sessions.py:54-63`  
**Impact:** With the new error isolation, if `grant_xp` fails, `xp` stays 0 and the coaching prompt receives `xp_earned=0` instead of the actual value. This is cosmetic (the coaching message is wrong) but doesn't cause data corruption.

**Recommendation:** Compute `xp_for_session` outside the try block since it's a pure calculation, and only wrap the DB-writing `grant_xp` call.

### L-2 — `coaching_engine.py` `_store_message` commits after each message
**Severity:** Low  
**Files:** `backend/app/services/coaching_engine.py:70-85`  
**Impact:** `_store_message` does `db.commit()` + `db.refresh()`, which is fine for its own scope but means partial state is committed even if the caller's broader transaction needs to roll back. Currently not an issue since there's no broader transaction wrapping coaching.

## Changes Made

| Finding | File | Change |
|---------|------|--------|
| H-1 | `backend/app/routers/sessions.py` | Background task now creates its own `SessionLocal()` with `try/finally: db.close()` |
| H-2 | `backend/app/routers/sessions.py` | Each completion step wrapped in individual `try/except` with structured logging |
| M-1 | `backend/app/routers/sessions.py` | `completed_at` auto-set to `datetime.utcnow()` when status transitions to `"completed"` |

## Verification

- Pattern consistency: Background task DB session lifecycle now matches scheduler jobs in `main.py`

## Remaining Recommendations
1. **M-2 idempotency**: Add a DB-level check or `SELECT FOR UPDATE` to prevent double-completion under concurrent load
2. **L-1 XP isolation**: Move `xp_for_session()` outside the `try` block so coaching gets the correct XP value even if `grant_xp` fails
3. **Test coverage**: Add integration tests for the completion pipeline, especially the error isolation and DB lifecycle
