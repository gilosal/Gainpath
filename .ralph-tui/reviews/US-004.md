# US-004: Audit and Harden AI Client Fallback Behavior

**Date:** 2026-04-18  
**Scope:** `backend/app/services/ai_client.py`, `backend/app/routers/coaching.py`, `backend/app/main.py`  
**Status:** Complete

---

## Findings (Severity-Ranked)

### S-1 — `tenacity` `reraise=False` swallowed retryable errors in `generate()` path
**Severity:** High  
**File:** `backend/app/services/ai_client.py:170-171`  
**Impact:** When `_call_with_retry` was decorated with `@retry(..., reraise=False)` and the retried exception type was `_RETRYABLE`, the last attempt's exception was silently swallowed by tenacity, and the `except` block below it would never catch the real cause. The method would proceed into the JSON-parsing `try` block with stale state or fall through without ever raising the original network error, leaving callers (especially `generate()`) with an opaque `AIGenerationError("No models configured.")` that hides the real cause.  
**Fix:** Changed to `reraise=True` so transient network errors propagate correctly after retries are exhausted, and added an explicit `except _RETRYABLE: raise` in the inner try/except to ensure retryable exceptions escape the broader catch-all.

### S-2 — `generate_text()` had no retry on transient network errors
**Severity:** High  
**File:** `backend/app/services/ai_client.py:50-121` (original)  
**Impact:** The `generate_text()` method (used for coaching chat) made a single raw HTTP call per model with no retry logic. A transient timeout or connection error would immediately fall through to the fallback model, halving resilience. The structured `generate()` method already had tenacity retries.  
**Fix:** Extracted HTTP logic into `_call_text_with_retry()` decorated with the same tenacity retry policy, so both codepaths now get up to `ai_max_retries` attempts before falling back.

### S-3 — Empty AI responses not detected — empty-string content could bubble up
**Severity:** Medium  
**File:** `backend/app/services/ai_client.py:99-100` (original)  
**Impact:** If the model returned `{"choices": [{"message": {"content": ""}}]}` or `{"content": "  "}`, both `generate_text` and `generate` would accept the empty string as valid. For structured JSON mode, an empty string would cause a `JSONDecodeError` with an unhelpful message. For text mode, empty content would be stored as a coaching message or chat response.  
**Fix:** Added `AIEmptyResponseError` and explicit empty-content guards in both `_call_with_retry` and `generate_text`, so empty responses are logged as failures and trigger fallback.

### S-4 — Markdown code fences broke `response_format: json_object`
**Severity:** Medium  
**File:** `backend/app/services/ai_client.py:200` (original)  
**Impact:** Some models wrap JSON in `` ```json\n...\n``` `` fences even when `response_format: json_object` is set. The original code did `json.loads(raw_content)` directly, which would fail on fenced content. This was a common source of `JSONDecodeError` that would surface as opaque `AIGenerationError`.  
**Fix:** Added `strip_json_fences()` helper that strips markdown code fences before `json.loads()`, matching a well-known provider quirk.

### S-5 — `generate_text()` did not log token usage or request IDs
**Severity:** Medium  
**File:** `backend/app/services/ai_client.py:94-121` (original)  
**Impact:** The original `generate_text` method logged `usage={}` and `request_id=None` for every call because it extracted only the content string from the response, discarding `usage` and `id` fields. The `ai_usage_log` table had no visibility into chat/text generation costs.  
**Fix:** Refactored `_call_text_with_retry` to return `(content, usage_data, request_id)` so `generate_text` can log real token counts and request IDs, matching the structured `generate()` path.

### S-6 — Coaching background tasks used request-scoped DB session
**Severity:** High  
**File:** `backend/app/routers/coaching.py:117-143` (original)  
**Impact:** All three coaching generation endpoints (`/generate/daily`, `/generate/weekly-summary`, `/generate/post-workout`) passed `db: Session = Depends(get_db)` into `BackgroundTasks.add_task()`. FastAPI closes the request-scoped session after the response, so the background task would operate on a closed session, causing `DetachedInstanceError` or silent failures.  
**Fix:** Removed `db` dependency from background task registration; each task wrapper now creates its own `SessionLocal()` with proper cleanup. Matches the pattern from US-002 for `sessions.py`.

### S-7 — Scheduler jobs lacked exception isolation
**Severity:** Low  
**File:** `backend/app/main.py:108-121` (original)  
**Impact:** The APScheduler jobs `_daily_motivation_job` and `_weekly_summary_job` had only `try/finally: db.close()` with no exception handler. An unhandled exception would propagate to the scheduler, potentially crashing the scheduler thread.  
**Fix:** Added `except Exception` with `logger.exception()` so failures are logged without killing the scheduler.

## Changes Made

| File | Change |
|------|--------|
| `backend/app/services/ai_client.py` | Fixed `reraise=False` → `reraise=True` on tenacity decorator; added `AIEmptyResponseError` with empty-content guards; added `strip_json_fences()` for markdown-fenced JSON; extracted `_call_text_with_retry` with tenacity retry and full usage/logging metadata; added `except _RETRYABLE: raise` to prevent retryable errors being caught by the broader handler |
| `backend/app/routers/coaching.py` | Fixed background task DB lifecycle: all three generation endpoints now use wrapper functions that create their own `SessionLocal()` instead of passing the request-scoped `db`; added try/except logging in each wrapper |
| `backend/app/main.py` | Added exception logging to scheduler jobs `_daily_motivation_job` and `_weekly_summary_job` |

## Not Changed (Intentional)

- **Coaching engine error handling** (`coaching_engine.py`): Each generation function already wraps `ai_client.generate()` in `try/except` returning `None` on failure — this is correct graceful degradation and was left unchanged.
- **Chat fallback message** (`coaching_engine.py:409`): The `chat()` function already catches exceptions and returns a friendly fallback string — this is correct and was left unchanged.
- **`ai_max_retries` config default (3)**: Already defined in `config.py` at `ai_max_retries: int = 3`; the tenacity decorators now reference `settings.ai_max_retries` instead of the hardcoded `3`.
