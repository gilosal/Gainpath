# US-005: Architecture Review — Gainpath / PaceForge

**Date:** 2026-04-18  
**Scope:** Frontend boundaries, backend layering, data flow, auth boundaries, deployment assumptions  
**Hot paths traced:** Today flow, session completion pipeline, AI coaching, plan generation

---

## Executive Summary

The codebase follows a clean single-user self-hosted app pattern with good separation between frontend and backend via a Next.js proxy. The biggest risks are: (1) localStorage auth is XSS-exposable, (2) several background-task paths silently swallow errors or lose data, (3) no input validation at the ORM boundary, and (4) no database migrations strategy surfaced in the deployment flow. The architecture is fundamentally sound for a single-user app — the debt items below are ranked by severity and impact on reliability.

---

## Severity-Ranked Findings

### H-1: Auth credential stored in localStorage (XSS-exposable)

**Files:** `frontend/lib/api.ts:40-44`, `frontend/lib/api.ts:46-48`

The password is stored in `localStorage` as plaintext (`paceforge-password`) and base64-encoded into every request's `Authorization` header. Any XSS vector (third-party script, compromised dependency, or future template bug) can exfiltrate the password immediately. The proxy already strips `WWW-Authenticate` — good — but the stored password itself remains the gap.

**Recommendation (short-term):** Add a `Content-Security-Policy` header via Next.js middleware to harden against script injection. Audit all `dangerouslySetInnerHTML` usage.  
**Recommendation (medium-term):** Migrate to an `HttpOnly` cookie-based session. Add a `/auth/login` endpoint that issues a signed JWT or session token stored as an `HttpOnly`/`Secure`/`SameSite=Strict` cookie. Remove `localStorage` credential handling entirely.

---

### H-2: No input validation at the ORM boundary

**Files:** `backend/app/routers/sessions.py:128-133`, `backend/app/routers/sessions.py:156-158`, `backend/app/routers/profile.py:28-33`, `backend/app/routers/offline.py:67-99`

Multiple endpoints use `payload.model_dump(exclude_unset=True)` → `setattr(log, field, value)` loops that apply any field the client sends directly to the ORM model. The Pydantic schemas (`SessionLogUpdate`, `UserProfileUpdate`, etc.) likely validate types but may not validate ranges, allowed values, or immutable fields. The `update_session` PATCH endpoint allows setting any field including `id` or `status` transitions that shouldn't be possible (e.g., going from "completed" back to "planned"). The offline sync `_apply_action` uses `**{k: v for k, v in payload.items()}` which spreads arbitrary keys.

**Recommendation (short-term):** Audit all Pydantic schemas for field-level constraints (min/max for RPE, allowed status transitions, immutable fields like `id`). Add explicit `status` transition validation in `update_session`.  
**Recommendation (medium-term):** Add a middleware or dependency that validates request payloads against an allowlist of fields per endpoint, rejecting unknown keys.

---

### H-3: Background task error isolation could lose XP and achievements silently

**Files:** `backend/app/routers/sessions.py:22-81`, `backend/app/routers/coaching.py:117-148`

The `_on_session_completed` background task correctly isolates each sub-step (PR detection, streak, XP, achievements, coaching) in its own `try/except`. However, the errors are only `log.exception()` — there is no retry mechanism and no user-facing feedback. If the PR detection step fails, the user never sees their new PR, and there's no way to re-trigger it. Similarly, `coaching.py` background task wrappers silently swallow all errors.

**Recommendation (short-term):** Add a `/sessions/{id}/reprocess` endpoint or "retry" button that re-runs the completion pipeline for a given session. Store failure state on the session log (e.g., `processing_status` field) so the UI can surface errors.  
**Recommendation (medium-term):** Replace `BackgroundTasks` with a proper task queue (e.g., Celery or `arq`) that supports retries and dead-letter queues.

---

### H-4: No database migration enforcement in deployment

**Files:** `docker-compose.yml:37-38`, `backend/alembic/` (referenced but not verified in startup)

The Docker entrypoint runs `alembic upgrade head` before `uvicorn`, which is correct. However, there is no health check on the backend container verifying that migrations completed successfully. A failed migration will crash the container, but a partially-applied migration could leave the database in an inconsistent state. There is also no `downgrade` safety net or migration lock.

**Recommendation (short-term):** Add a migration health check or startup health endpoint that confirms the database schema version matches the application's expected version.  
**Recommendation (medium-term):** Add migration rollback support and a CI check that validates migrations against a test database before deploying.

### M-1: Single-row user profile pattern has no concurrency guard

**Files:** `backend/app/models/user.py`, `backend/app/routers/profile.py:11-19`

The app assumes a single user via `_get_or_create_profile(db).first()`. Under concurrent requests (e.g., multiple frontend tabs), two `PUT /profile` calls could race — one could overwrite the other's changes. Similarly, `StreakSnapshot` is always `.first()` with no row-level locking.

**Recommendation (short-term):** Add `SELECT FOR UPDATE` when reading the profile or streak snapshot in write endpoints, or use optimistic locking with an `updated_at` check.  
**Recommendation (medium-term):** If multi-user support is ever planned, refactor to proper user-scoped queries with auth-derived user IDs.

---

### M-2: Dashboard endpoints do N+1 queries and load full row sets into memory

**Files:** `backend/app/routers/dashboard.py:18-72`, `backend/app/routers/dashboard.py:76-94`

`get_dashboard_summary()` loads all week's sessions into Python and iterates. The trend endpoints load up to 52 weeks of sessions and aggregate in Python. `exercise_progression` loads all matching `SetLog` rows and maps in Python. These will degrade as data grows.

**Recommendation (short-term):** Convert aggregations (`sum`, `count`, `group_by`) to SQL-level queries using `func.sum()` and `func.count()`.  
**Recommendation (medium-term):** Add database indexes on `session_log(session_date, status, session_type)` and `set_log(exercise_name, completed_at)`.

---

### M-3: `today` endpoint auto-creates session stubs from planned sessions

**Files:** `backend/app/routers/sessions.py:101-124`

`GET /sessions/today` has a side effect: if no session logs exist for today, it creates stubs from `PlannedSession` rows. This means a simple `GET` request is not idempotent and can create duplicate stubs if called multiple times before sessions are created (though `planned_session_id` uniqueness isn't enforced). It also means the frontend must handle partially-initialized state.

**Recommendation (short-term):** Add a `UNIQUE` constraint on `session_log(planned_session_id)` to prevent duplicate stubs. Make the stub creation conditional on not already existing.  
**Recommendation (medium-term):** Separate session stub creation into its own explicit `POST /sessions/from-plan` endpoint so `GET` remains side-effect-free.

### M-4: No CORS origin validation for production

**Files:** `backend/app/config.py:67`, `frontend/app/api/proxy/[...path]/route.ts:3`

`cors_origins` defaults to `["http://localhost:3000", "http://localhost:3001"]` and is loaded from `.env`. In production deployment, if `CORS_ORIGINS` is misconfigured or set to `["*"]`, the API would accept calls from any origin. Since auth is HTTP Basic, this would be a direct credential leak. The proxy approach mitigates this (frontend calls proxy, proxy strips `WWW-Authenticate`), but the backend is still directly accessible on port 8000.

**Recommendation (short-term):** Validate `cors_origins` at startup — reject `["*"]` if `APP_ENV != "development"`.  
**Recommendation (medium-term):** In production, bind the backend to `127.0.0.1` or use a reverse proxy that doesn't expose port 8000 publicly.

### M-5: AI client creates a new `httpx.AsyncClient` per request

**Files:** `backend/app/services/ai_client.py:287-294`

Every call to `_http_post` creates a new `httpx.AsyncClient(timeout=...)`, makes one request, and discards it. This means TCP connection establishment, TLS handshake, and connection teardown happen per AI call. For a gym app making frequent coaching calls, this adds latency and resource waste.

**Recommendation (short-term):** Create the `httpx.AsyncClient` once in `AIClient.__init__` or as a module-level singleton, and reuse it. Add an `async def close()` method or use lifespan events.  
**Recommendation (medium-term):** Consider adding connection pooling configuration (`limits=httpx.Limits(max_connections=10)`).

---

### M-6: Scheduler uses deprecated `@app.on_event("startup")`

**Files:** `backend/app/main.py:64-68`

FastAPI's `on_event` handlers are deprecated in favor of the `lifespan` context manager pattern. The scheduler also has no graceful shutdown — `AsyncIOScheduler` is started but never shut down, which can cause hanging on process exit.

**Recommendation (short-term):** Migrate to `lifespan` context manager. Store the scheduler instance and call `scheduler.shutdown()` in the shutdown handler.  
**Recommendation (medium-term):** Move scheduler initialization into a dedicated module with proper lifecycle management.

---

### M-7: Frontend offline sync delegates entirely to backend

**Files:** `frontend/lib/offline-queue.ts:89-107`, `backend/app/routers/offline.py:36-99`

The `sync()` method calls `offlineApi.sync()` which processes the server-side offline queue. But the client-side IndexedDB queue's `logSetWithFallback` enqueues actions with full payloads that include arbitrary keys — the backend's `_apply_action` spreads `payload.items()` directly into ORM models. This is an amplification of H-2: offline sync has no payload shape validation at all.

**Recommendation (short-term):** Add explicit Pydantic models for each offline action type (`CreateSetLogPayload`, `CompleteSessionPayload`, `AddBodyFeedbackPayload`) and validate payloads against them before ORM insertion.  
**Recommendation (medium-term):** Consider an append-only event log pattern for offline writes, where actions are recorded as events and processed idempotently.

### L-1: No pagination on several list endpoints

**Files:** `backend/app/routers/sessions.py:86-98` (has `limit`), `backend/app/routers/gamification.py:127-150`, `backend/app/routers/coaching.py:50-70`

`sessions.list_sessions` has `limit=50` but no `offset`/cursor. `coaching.list_messages` has `limit=20` but no cursor. `gamification.achievements` and `gamification.prs` have `limit=100` but no proper pagination. As data grows, these will return increasingly large payloads.

**Recommendation (short-term):** Add `offset` parameter to all list endpoints and document maximum page sizes.  
**Recommendation (medium-term):** Implement cursor-based pagination for time-series data (sessions, coaching messages).

---

### L-2: `datetime.utcnow()` is deprecated in Python 3.12+

**Files:** `backend/app/routers/sessions.py:161`, `backend/app/services/achievement_engine.py:222`, `backend/app/models/gamification.py:12`, `backend/app/services/streak_engine.py:69,99,175`, and others

Multiple files use `datetime.utcnow()` which is deprecated in Python 3.12+ in favor of `datetime.now(datetime.timezone.utc)`.

**Recommendation (short-term):** Replace `datetime.utcnow()` with `datetime.now(timezone.utc)` across all files.  
**Recommendation (medium-term):** Create a `now_utc()` helper in a shared utility module.

---

### L-3: Frontend has no error boundary or loading state convention

**Files:** `frontend/components/today/TodayView.tsx`, `frontend/components/today/ActiveWorkout.tsx`

The `useQuery` calls handle loading with skeleton cards but have no error state rendering. If the API returns an error, React Query will silently show the loading skeleton indefinitely (or stale data). The `useMutation` hooks (complete, skip) have no `onError` callback — failures are invisible to the user.

**Recommendation (short-term):** Add `error` state handling to all `useQuery` calls with a retry button or toast notification. Add `onError` callbacks to mutations.  
**Recommendation (medium-term):** Create an `ErrorBoundary` component and a convention for error states in the design system.

---

### L-4: Chat endpoint stores user message before AI response, creating orphan rows on failure

**Files:** `backend/app/services/coaching_engine.py:397-415`

The `chat()` function calls `db.add(ChatMessage(...role="user"))` and `db.commit()` before calling `ai_client.generate_text()`. If the AI call fails, the user message is already committed. The `except` block returns a fallback string but doesn't store it, so the ChatMessage table has the user message but no assistant response. The coaching router then queries the last 2 messages and indexes `[1]` as `assistant_msg`, which will crash with an `IndexError` if there's only one message.

**Recommendation (short-term):** Store both messages in a single transaction: generate the AI response first, then commit user + assistant message together. Or store the fallback message in the `except` block.  
**Recommendation (medium-term):** Add a `status` field to `ChatMessage` (`pending`, `completed`, `failed`) so orphan messages can be identified and retried.

---

### L-5: Hardcoded `PaceForge` branding in AI client and backend

**Files:** `backend/app/services/ai_client.py:273-274`, `backend/app/config.py:65`

The `X-Title` header in AI requests is hardcoded to `"PaceForge"`. The `app_name` config defaults to `"PaceForge"`. The frontend localStorage key is `paceforge-password`. If the product name changes, these would need to be updated in disparate places.

**Recommendation (short-term):** Replace hardcoded `"PaceForge"` in AI client headers with `settings.app_name`.  
**Recommendation (medium-term):** Centralize all branded strings into a single config or environment variable.

## Data Flow Summary

```
┌─────────────────────────────────────────────────────────┐
│                    Frontend (Next.js)                     │
│   TodayView ──→ ActiveWorkout / QuickLogSheet              │
│       │                                                   │
│       ├── sessionsApi.* ──→ fetch(/api/proxy/...)         │
│       ├── gamificationApi.* ──→                           │
│       ├── coachingApi.* ──→                               │
│       └── offlineApi.* ──→                                │
│                 │                                         │
│    api.ts (Basic auth from localStorage)                  │
│                 │                                         │
├─────────────────┼───────────────────────────────────────┤
│    Next.js Proxy Route (/api/proxy/[...path])             │
│    → forwards to backend, strips WWW-Authenticate          │
├─────────────────┼───────────────────────────────────────┤
│                 ▼                                         │
│   FastAPI Backend (:8000)                                 │
│       ├── verify_password (Depends on all routers)        │
│       ├── Routers: sessions, plans, coaching,             │
│       │             gamification, dashboard, offline       │
│       ├── Services: ai_client, coaching_engine,           │
│       │              pr_detector, streak_engine,          │
│       │              achievement_engine                    │
│       └── Models: session, plan, user, gamification,      │
│                   coaching, ai_usage                       │
│                  │                                        │
│                  ▼                                        │
│           PostgreSQL (:5432)                              │
└─────────────────────────────────────────────────────────┘
```

**Auth flow:** Frontend stores password in `localStorage` → attaches as `Authorization: Basic ...` header → Next.js proxy forwards to FastAPI → `verify_password` dependency validates against `settings.app_password` using `secrets.compare_digest`.

**Session completion flow:** `PATCH /sessions/{id}` with `status: "completed"` → sets `completed_at` → commits → `BackgroundTasks.add_task(_on_session_completed)` → background task creates own DB session → sequentially runs PR detection, streak update, XP grant, achievement check, post-workout coaching → each isolated in try/except.

**AI coaching flow:** Scheduler (cron) or manual trigger → `BackgroundTasks` → `coaching_engine.generate_*` → `ai_client.generate()` → OpenRouter API (primary + fallback model) → Pydantic validation → store as `CoachingMessage`.

## Architecture Strengths

1. **Clean proxy boundary** — Next.js proxy strips auth headers, preventing browser-native Basic auth prompts.
2. **Centralized AI client** with retry, fallback model, usage logging, and empty-response detection.
3. **Background task DB lifecycle** — Correct `SessionLocal()` pattern in background tasks (fixed in US-002/US-004).
4. **Offline-first** — IndexedDB queue with `logSetWithFallback` for spotty gym Wi-Fi.
5. **Isolated side effects** — Session completion pipeline isolates each step so one failure doesn't block others.
6. **Startup config validation** — Refuses to boot with insecure defaults in production.
